2.1 Порівняння Hadoop з реляційними і NoSQL базами даних

Що означає NoSQL?

  • Не те, що SQL не повинен використовуватися для обробки великих даних.
  • Не те, що SQL та реляційні БД застаріли та відмирають.
  • Не лише SQL. Для певного кола завдань інші підходи до зберігання та обробки інформації можуть бути ефективнішими, особливо з точки зору легкості розширення (масштабуемості) інформаційної бази при постійному додаванні нових даних.
  • Не плутати з NewSQL – новий підхід до побудови реляційних БД для додатків, орієнтованих на обробку великої кількості транзакцій, що поєднує зручності SQL із масштабуванням NoSQL систем.

  • Готовність поступитися цілісністю задля доступності.
  • Дешеве/безкоштовне ПЗ.
  • Не треба знати мови, крім стандартних Java/C/C++.
  • Швидший доступ до даних.

Однією із ключових технологій реалізації концепцій Big Data є платформа Hadoop.

Hadoop– це програмна платформа (Software Framework) побудови розподілених програм для масово-паралельної обробки (Massive Parallel Processing, MPP) даних. У платформі Hadoop умовно можна виділити 2 основних компоненти:

  • Hadoop Distributed File System(HDFS) – розподілена файлова система, яка забезпечує високошвидкісний доступ до даних програми;
  • MapReduce– програмна платформа для розподіленої обробки великих обсягів даних на обчислювальному кластері.
  • Що таке Hadoop?

  • • Open source (проект Apache Software Foundation)

  • • Служить для пакетної обробки та ETL
  • • Обробляє дані у масивно-паралельному режимі
  • • Працює на дуже великих кластерах (від сотень до тисяч вузлів) на дешевому «залізі»
  • • Автоматично обробляє відмову вузлів, та перерозподіл даних
  • • Використовується у багатьох відомих проектах

До основних технічних характеристик платформи Hadoop відносять:

  • Масштабованість: платформа масштабується лінійно і дозволяє зберігати та обробляти петабайти даних;
  • Стійкість до збоїв: всі дані, що зберігаються, надмірні, всі провалені завдання з обробки даних перезапускаються;
  • Кросплатформеність: бібліотеки Hadoop написані (в основному) на Java і можуть виконуватися в будь-якій операційній системі, що підтримує JVM (Java VM);
  • Автоматичне розпаралелювання виконання завдання: Hadoop створює «чисті» абстракції для розробників, знімаючи з них роботу з планування, контролю та агрегатування результатів паралельної обробки даних.

Бізнес-вигоди від використання Hadoop можна визначити як:

  • Гнучкість: зберігання та аналіз структурованих та неструктурованих типів даних;
  • Ефективність: у більшості випадків нижча вартість зберігання/обробки терабайта даних порівняно з існуючими рішеннями;
  • Низька вартість створення кластера: для створення Hadoop-кластера не потрібне дороге серверне апаратне забезпечення.
  • Порівняльна легкість адаптації: Hadoop має широку екосистему, що активно розвивається;
  • Мінімальні ризики пов'язані з некоректною роботою ядра платформи: на сьогоднішній день платформа Hadoop успішно використовується для обробки петабайт інформації;
  • "Open Source" ліцензування: низька вартість впровадження та володіння платформою Hadoop, велике «developer community».

Різниця між Hadoop і SQL

Фреймворк Hadoop проти SQL

– В основі екосистеми Hadoop є два х компоненти – розподілена файлова система Hadoop (HDFS) – розподілена, масштабована та портативна файлова система, написана мовою Java для зберігання дуже великих наборів даних у кластерах комп’ютерів; і підхід до розподіленої обробки на основі Java під назвою MapReduce. З іншого боку, SQL Server — це реляційна система керування базами даних і одна з найпотужніших у світі платформ даних, яка використовується рядом комерційних і власних продуктів для запитів, обробки та візуалізації різноманітних джерел даних.

Тип даних

– Hadoop розроблено для роботи з будь-якими типами даних, структурованими, напівструктурованими чи неструктурованими, що робить його дуже гнучким для роботи, коли справа стосується обробки великих даних. SQL, з іншого боку, є мова програмування спеціально створена для керування та запитів даних у системах керування реляційними базами даних (RDBMS). Вона базується на моделі сутності та зв’язку RDBMS, тому може обробляти лише структуровані дані. SQL не можна використовувати для неструктурованих даних, оскільки вони не відповідають моделі даних без легко ідентифікованої структури.

Обробка

– HDFS — це розподілена файлова система, розроблена для підтримки пакетної обробки даних, тобто дані збираються пакетами, і кожен пакет надсилається на обробку. Пакет може тривати від одного дня до однієї хвилини. Оскільки він призначений для пакетної обробки, він не має концепція випадкового читання або запису. Навпаки, SQL Server як платформа бази даних загального призначення підтримує обробку даних у реальному часі, тобто дані передаються від відправника до одержувача, як тільки вони створюються на кінці джерела.

Продуктивність Hadoop і SQL

  • – Архітектура Hadoop іноді призводить до невідповідності імпедансу між зберіганням даних і доступом до них. Він має менше обмежень або перевірок щодо даних, які він зберігає, і він не має тих самих можливостей кінцевого користувача та екосистеми, які розробив SQL. З іншого боку, SQL Server справляється з забезпеченням якості та узгодженості даних набагато краще, ніж Hadoop, що дає змогу використовувати екосистему інструментів аналізу та візуалізації даних на основі SQL. Однак SQL також має деякі недоліки, серед яких масштабованість для обробки величезних обсягів даних і підтримка зберігання даних у вільному форматі.

Підсумок Hadoop проти SQL

Hadoop є найбільш популярним і широко поширеним інструментом Big Data, розробленим для роботи з будь-якими типами даних – структурованими, неструктурованими чи напівструктурованими. Але коли мова заходить про RDBMS, SQL є, мабуть, найпотужнішою системою зберігання та керування даними, що містить динамічну оперативну пам’ять. Однак існуючі рішення RDBMS, такі як SQL Servers, призначені лише для керування значним обсягом даних, але не для неструктурованих або напівструктурованих даних зі змінними атрибутами. Як і в багатьох платформах, Hadoop і SQL Server мають достатню частку сильних і слабких сторін. Використовуйте обидва разом, і ви зможете використовувати сильні сторони кожногопокипом'якшення слабких місць.

Екосистема платформи Hadoop

Центральне місце екосистеми Hadoop займає сховище даних (Data Storage). Hadoop підтримує зберігання як неструктурованих даних із розподіленою файловою системою HDFS, так і структурованих даних у нереляційній базі даних HBase.

Нижче коротко перераховані проекти Hadoop.

Avro— система серіалізації для виконання ефективних міжмовних викликів RPC та довгострокового зберігання даних.

MapReduce- модель розподіленої обробки даних та виконавче середовище, що працює на великих кластерах типових машин.

HDFS- Розподілена файлова система, що працює на великих кластерах стандартних машин.

Pig- мова управління потоком даних і виконавче середовище для аналізу великих наборів даних. Pig працює в HDFS та кластерах MapReduce.

Hive- Розподілене сховище даних. Hive керує даними, що зберігаються в HDFS, і надає мову запитів на базі SQL (які перетворюються ядром часу виконання на завдання MapReduce) для роботи з цими даними.

HBase- Розподілена стовпцово-орієнтована база даних. HBase використовує HDFS для організації зберігання даних і підтримує пакетні обчислення з використанням MapReduce, так і точкові запити (довільне читання даних).

ZooKeeper- Розподілений координаційний сервіс високої доступності. ZooKeeper надає примітиви, які можна використовувати для побудови розподілених додатків (наприклад, розподілені блокування).

Sqoop— інструмент ефективного масового пересилання даних між структурованими сховищами (такими як реляційні бази даних) та HDFS.

Oozie— сервіс запуску та планування завдань Hadoop (включаючи завдання MapReduce)

2.4 Робота з HDFS

Файлова система HDFS спроектована для зберігання дуже великих файлів з потокової схемою доступу до даних в кластерах звичайних машин1. Розглянемо це твердження більш докладно.

Дуже великі файли

Під «дуже великими» в цьому контексті маються на увазі файли, розмір яких складає сотні мегабайт, гігабайт і терабайт. Зараз існують кластери Hadoop, в яких зберігаються петабайт данних2.

Потоковий доступ до даних

В основу HDFS закладена концепція однократного запису / багаторазового читання як найефективніша схема обробки даних. Набір даних зазвичай генерується або копіюється з джерела, після чого з ним виконуються різні аналітичні операції. У кожній операції задіюється велика частина набору даних (або весь набір), тому час читання всього набору даних важливіше затримки читання першого запису.

звичайне обладнання

Hadoop не вимагає дорогого устаткування високої надійності. Система спроектована для роботи на стандартному обладнанні (загальнодоступне обладнання, яке може бути придбано у багатьох фірм) з досить високою ймовірністю відмови окремих вузлів в кластері (по крайней мере, для великих кластерів). Технологія HDFS спроектована таким чином, щоб у разі відмови система продовжувала роботу без скільки-небудь помітного переривання.

Також слід виділити області застосування, для яких в даний час HDFS підходить не найкращим чином (при тому, що в майбутньому ситуація може змінитися):

Швидкий доступ до даних

Додатки, що вимагають доступу до даних з мінімальною затримкою (в діапазоні десятків мілісекунд), погано поєднуються з HDFS. Нагадаємо, що система HDFS оптимізована для забезпечення високої пропускної здатності передачі даних, за яку доводиться розплачуватися уповільненням доступу. HBase в даний час краще підходить для організації доступу до даних з мінімальною затримкою.

Численні дрібні файли

Так як вузол імен зберігає метадані файлової системи в пам'яті, межа кількості файлів в файлову систему визначається обсягом пам'яті вузла імен. Як показує досвід, кожен файл, каталог і блок займають близько 150 байт. Таким чином, наприклад, якщо у вас є мільйон файлів, кожен з яких займає один блок, для зберігання інформації буде потрібно не менше 300 Мбайт пам'яті. Зберігання мільйонів файлів ще прийнятно, але мільярди файлів вже виходять за межі можливостей сучасного встаткування1.

Множинні джерела записи, довільні модифікації файлів

Запис в файли HDFS може виконуватися тільки одним джерелом. Запис завжди здійснюється в кінець файлу. Підтримка множинних джерел записи або модифікації з довільним зміщенням в файлі відсутній. (Може бути, ці можливості будуть підтримуватися в майбутньому, але, швидше за все, вони будуть відносно неефективними).

Основні концепції HDFS

блоки

З дисковим пристроєм зв'язується розмір блоку - мінімальний обсяг даних, які використовуються в операції читання або запису. Однодискові файлові системи використовують цю обставину, повертаючи дані в блоках, розмір яких кратний розміру дискового блоку. Розмір блоку файлової системи зазвичай становить кілька кілобайт, тоді як розмір дискового блоку зазвичай дорівнює 512 байтам. Як правило, всі ці нюанси повністю прозорі для користувачів файлових систем, які просто читають або записують в файли дані довільної довжини. Однак програми супроводу файлових систем (такі, якdf і fsck) Працюють на рівні блоків файлової системи.

У HDFS теж існує концепція блоку, але він має істотно більший розмір - за замовчуванням 94 Мбайт. Як і в однодискових файлових системах, файли HDFS розбиваються на блокові фрагменти, збережені незалежно один від одного. На відміну від однодискових файлових систем, файл HDFS, менший одного блоку, не займає весь простір, виділене під блок. Це зроблено для того, щоб звести до мінімуму кількість операцій позиціонування. За замовчуванням використовується значення 94 Мбайт, хоча в багатьох установках HDFS розмір блоку збільшений до 128 Мбайт.

Абстракція блоків в розподіленої файлової системи має кілька переваг.

перша перевага найбільш очевидно: файл може бути більше будь-якого окремого диска в мережі. Блоки файлу зовсім необов'язково зберігати на одному диску; вони можуть використовувати будь-які диски в кластері. Більш того, при зберіганні файлу в кластері HDFS його блоки можуть бути розподілені по всіх дисках кластера (хоча ця ситуація дещо нетипова).

Друге:використання в якості абстрактної одиниці блоку замість файлу спрощує підсистему зберігання. Простота - бажане властивість будь-яких систем, але вона особливо важлива в розподілених системах з їх різноманітними можливими режимами відмов. Використання блоків в підсистемі зберігання даних спрощує зберігання (так як блоки мають фіксований розмір, система може легко обчислити кількість блоків, які можуть зберігатися на заданому диску) і усуває проблеми з метаданими (блок - всього лише фрагмент даних, призначений для збереження, з ним не потрібно зберігати метадані файлу - скажімо, інформацію про дозволи доступу, які можуть окремо оброблятися іншою системою).

Крім того, блоки добре вписуються в механізм реплікації - вони покращують відмовостійкість і доступність системи. Для захисту від пошкоджених блоків і збоїв дисків / машин, кожен блок реплікується на невеликій кількості фізично розділених машин (як правило, трьох). Якщо блок стає недоступним, його копія читається з іншого місця, причому це відбувається прозоро для клієнта. Блок, який став недоступним через пошкодження даних або збою обладнання, копіюється з альтернативних сховищ на інші працездатні машини, щоб довести фактор реплікації до нормального рівня. Крім того, деякі програми можуть встановити високий фактор реплікації для блоків часто запитуваної файлу, щоб поліпшити розподіл навантаження по читанню в межах кластера.

Вузли імен та вузли даних

Кластер HDFS складається з двох типів вузлів, що працюють за схемою «управляющій- підлеглий»: вузол імен (управитель) і кілька вузлів даних (підлеглі). Вузол імен керує простором імен файлової системи. Він підтримує дерево файлової системи і метадані всіх файлів і каталогів в дереві. Інформація зберігається на локальному диску у вигляді двох файлів: образу простору імен і журналу змін. Вузол імен також знає, на яких вузлах даних зберігаються всі блоки заданого файлу; проте інформація про місцезнаходження блоків не зберігається постійно, а будується заново за відомостями вузлів даних при запуску системи. Щоб звернутися до файлової системи за дорученням користувача, клієнт зв'язується з вузлом імен та вузлами даних. Так що для користувача коду не потрібно нічого знати про вузли імен та вузлах даних.

Вузли даних - основна «робоча сила» файлової системи. Вони читають і записують блоки (на вимогу клієнтів або вузла імен), а також періодично передають вузлу імен список зберігаються ними блоків.

Без вузла імен файлова система стає непрацездатною. Більш того, при знищенні машини, на якій працює вузол імен, всі файли в файлової системі будуть втрачені, тому що відновити їх по блокам вузлів даних буде неможливо. З цієї причини важливо зробити вузол імен стійким до можливих збоїв; Hadoop надає два механізму вирішення цього завдання.

перший спосіб- архівація файлів, що визначають стійкий стан метаданих файлової системи. Hadoop можна налаштувати таким чином, щоб вузол імен записував своє стійке стан в кілька файлових систем. Операції запису виконуються синхронно і атомарному. Зазвичай в конфігурації налаштовується запис на локальний диск і в віддалений тому NFS.

Також можна запустити вторинний вузол імен, який, незважаючи на назву, не виконує функції вузла імен. Його основна функція - періодичне злиття образу простору імен з журналом змін, щоб запобігти надмірне розростання останнього. Вторинний вузол імен зазвичай працює на окремій фізичній машині, тому що для виконання злиття необхідні значна обчислювальна потужність і стільки ж пам'яті, скільки на вузлі імен. Вторинний вузол імен зберігає копію об'єднаного образу простору імен, який може використовуватися в разі збою основного вузла. Проте, стан вторинного вузла імен дещо відстає від стану основного вузла, так що в разі повної відмови останнього втрата даних практично неминуча.

Зазвичай в таких випадках файли метаданих вузла імен копіюються з NFS на вторинний вузол, який починає працювати як новий основний.

Архітектура HDFS

Блок файлу в HDFS: Дані в HDFS завжди зберігаються в блоках. Таким чином, один блок даних розділений на кілька блоків розміром 128 МБ, який є стандартним, і можна змінити його вручну.

Реплікація забезпечує доступність даних. В Блоках файлів HDFS зберігає дані у формі різних блоків, у той же час Hadoop також налаштовано для створення копії цих блоків файлів.

За замовчуванням фактор реплікації для Hadoop встановлено на 3, що можна налаштувати, це означає, що ви можете змінити його вручну відповідно до ваших вимог, як у прикладі вище, ми створили 4 блоки файлів, що означає, що створено 3 репліки або копії кожного блоку файлів. всього 4×3 = 12 блоків зроблено для цілей резервного копіювання.

Це тому, що для запуску Hadoop використовується стандартне обладнання (недороге системне обладнання), яке може вийти з ладу в будь-який момент. Ось чому потрібна така функція в HDFS, яка може створювати копії цих блоків файлів для цілей резервного копіювання, це відомо як відмовостійкість.

Стійка — це не що інше, як фізична колекція вузлів у кластері Hadoop (можливо, від 30 до 40). Великий кластер Hadoop складається з деякої кількості стійок. Racks Namenode вибирає найближчий Datanode для досягнення максимальної продуктивності під час виконання читання/запису інформації, що зменшує мережевий трафік.
